一、Git的对象模型设计理念

1. 快照存储模式

快照流 模式，每次提交时，Git都会对当前项目的全部文件各创建一个快照。
同时，为了提升效率，对于没有修改的文件，Git不会重复存储，而是创建一个指向上一版本的链接。
可以把Git理解位一个内容寻址数据库
Git 的数据主要存在 .git/objects 目录中，以对象的方式进行存储，常见的对象有4种，每个对象都有一个唯一的哈希ID：

  • blob：保存文件内容，但是没有文件名，按照内容哈希
  • tree：保存目录结构，记录目录有哪些目录和文件，文件名字，对象哈希，文件权限和类型
  • commit：保存提交信息（条目）：tree <根目录tree哈希> parent <父提交哈希> author
  • tag：标签信息
  • branch： 指向某次提交的指针
    commit -> tree -> blob 一次提交 -> 一个根目录 -> 多个文件内容

2. 三种文件状态

Git 文件的三个状态其实就是同一份文件在三个位置里的不同版本

Git 的三个状态可以理解为：工作区、暂存区、版本库中各有一份文件快照，文件状态
的变化本质就是这三处快照是否一致

三个位置：

  • 工作区：Working Directory 实际文件
  • 暂存区：Index 准备提交的快照，一个索引文件，.git/index
  • 版本库：Repositry 已提交的 commit 快照，就是 .git/objects 等内容
    git add main.c 就是将main.c的内容生成一个新的 blob 对象，将index的 main.c项 指向这个 blob
    git commit 就是根据暂存区记录的内容生成一个新的 tree 再生成一个新的 commit
    提交后，当前 branch 会指向这个新的 commit

三个状态：

  • commited：三个区域的内容完全一致
  • modified：工作区和暂存区的不同
  • staged：更新了index
  • untracked：新建文件，没有加入git管理
  • deleted：Work中删除了 git rm a.txt
    modified 看 Work 和 Index 是否不同；
    staged 看 Index 和 HEAD 是否不同；
    committed 看 Work、Index、HEAD 是否都相同。

3. 对象模型解释Git

git add main.c

  1. 把 main.c 当前内容写入 .git/objects，生成 blob
  2. 更新 .git/index ，让 main.c 项指向这个 blob

git commit -m "update main"

  1. 根据 index 生成 tree 对象
  2. 根据 tree 生成 commite 对象
  3. 让当前分支指向新的 commit

git diff ： 比较 工作区 和 暂存区 看哪些修改没有被 git add
git diff --staged ：比较 暂存区和HEAD 看下一次commit会提交哪些修改？
git diff HEAD ：比较 工作区和HEAD 有哪些变化?

二、Git 回滚

Git 提供了许多回滚机制，每种方法适用于不同场景。

1. 工作目录级别的回滚

git restore [--staged] 命令进行回滚

  1. 撤销修改：当文件只在工作目录中被修改没有执行 git add 到暂存区，可以 git restore XXX将文件状态恢复到最近提交的状态
  2. 取消暂存：当文件被 git add 到暂存区后，但是不该暂存某些文件，可以 git restore --staged XX

2. 提交级别的回滚

git reset ：就是移动 HEAD 指针，会修改历史

  1. git reset --soft ：仅将当前 HEAD 移动到指定commit 对象
  2. git reset --mixed：默认：将 HEAD 和 index 都变为指定的 commit 状态
  3. git reset --hard： 将 HEAD INDEX 和 WORK 都变为指定的 commit 状态

git revert：用新的提交撤销旧的提交，也就是UNDO，新增反向提交，适合远程使用

3. 切换到分支和提交

git switch --detach <commit-id> ： 将 WORK、INDEX、HEAD 都变到对应状态

三、分支与合并

1. 分支

git branch <新分支> ：创建分支，分支就是一个指向某个 commit 的可移动指针通常在.git/refs/heads 下：

  1. 创建一个 branch 对象，branch 对象默认指向当前 HEAD 指向的 commit 对象
  2. 提交时，一般新创建一个 commit 对象，它的 parent 指向 HEAD指向的commit，然后HEAD指向它
  3. 变分支就相当于切换 HEAD

2. 合并

git merge <分支>：要想弄清除合并，要先弄清楚三个角色：

A -- B -- C ---- M  <- main
        \        /
         D -- E  <- feature

git switch main && git merge feature

  • 共同祖先：B
  • 当前分支：C
  • 要合并的分支：E

合并时：Git 要看 B -> C main 做了什么 B -> E feature 做了什么
然后尝试做三方合并，生成一个新的merge commit
合并提交有两个父提交：C、E

A -- B -- C ---- M  <- main
	\        /
	 D -- E  <- feature

3. 合并冲突

如果 B->C, B->E都修改了同一处，Git不知道最后的状态是什么。就冲突了。
执行后需要手动修改：

  • HEAD：C <- main
  • INDEX：冲突状态，记录了 base/ours/theirs 多个版本
  • WORK：带冲突标记的文件
  1. git status 查看冲突文件
  2. git add <file> 标记冲突解决
  3. git commit 完成 merge commit
  4. git merge --abort 放弃合并

4. 变基

rebase：把一条线搬到另一条线后面，也就是移动共同祖先base
让历史变为了线性。
只在个人分支上 rebase。共享分支一般不允许 rebase。会导致大量的冲突和重复提交。
